How Prisma Works with Database Connections
==========================================

1. PRISMA CLIENT INSTANTIATION (config/prisma.js)

   The PrismaClient is instantiated as a singleton pattern:

     const { PrismaClient } = require('@prisma/client');
     const prisma = new PrismaClient({ errorFormat: '...' });

   - In development, the instance is cached on the global object (global.prisma)
     to prevent multiple instances during hot-reloading with nodemon.
   - In production, a fresh instance is created and not cached globally.
   - This prevents exhausting database connections from creating new
     PrismaClient instances on every module reload.

2. CONNECTION LIFECYCLE

   a) Connection Pool
      - Prisma uses a connection pool (default: 10 connections for PostgreSQL).
      - When you call prisma.$connect(), it initializes the pool but does NOT
        immediately open all connections — connections are lazily created as
        queries arrive.
      - The pool is managed internally by the Prisma query engine.

   b) Connection on Startup (server.js:44-57)

        await prisma.$connect();
        console.log('PostgreSQL connected successfully');
        server = app.listen(port, ...);

      - $connect() validates that the database is reachable using DATABASE_URL.
      - If it fails, the server exits gracefully after disconnecting.
      - Once connected, subsequent queries reuse pooled connections.

   c) Graceful Shutdown (server.js:59-76)

        const shutdown = async (signal) => {
          if (server) { server.close(async () => { await prisma.$disconnect(); }); }
        };
        process.on('SIGINT', () => shutdown('SIGINT'));
        process.on('SIGTERM', () => shutdown('SIGTERM'));

      - On SIGINT (Ctrl+C) or SIGTERM (process kill), the HTTP server stops
        accepting new requests first, then prisma.$disconnect() closes all
        pool connections cleanly.
      - This prevents hanging connections and ensures data integrity.

3. DATASOURCE CONFIGURATION (schema.prisma)

     datasource db {
       provider = "postgresql"
       url      = env("DATABASE_URL")
     }

   - provider: Specifies PostgreSQL as the database dialect.
   - url: Reads from DATABASE_URL environment variable (set in .env).
   - The URL format: postgresql://USER:PASSWORD@HOST:PORT/DATABASE

4. CONNECTION STRING USED

     DATABASE_URL="postgresql://postgres:pranav123@localhost:5432/Ecom"

   - postgresql://postgres:pranav123@localhost:5432/Ecom
   - user: postgres, password: pranav123, host: localhost, port: 5432, db: Ecom

5. HOW QUERIES USE THE CONNECTION

   - Every time you call a Prisma Client method like:
       prisma.user.findMany()
       prisma.product.create({ data: {...} })
       prisma.order.findUnique({ where: { id } })
   - The Prisma Client internally:
     a) Serializes the query into SQL via the Prisma query engine.
     b) Acquires a connection from the pool.
     c) Executes the SQL against PostgreSQL.
     d) Returns the connection to the pool.
     e) Maps the result back to JavaScript objects with correct types.

6. CONNECTION POOL SETTINGS (Default for PostgreSQL)

   - Pool size: 10 connections (configurable via DATABASE_URL params).
   - Connection timeout: 30 seconds (default).
   - Idle timeout: Connections are kept alive and reused; idle connections
     are eventually cleaned up by PostgreSQL.
   - To customize: DATABASE_URL="postgresql://.../?connection_limit=5"

7. BEST PRACTICES OBSERVED IN THIS PROJECT

   - Singleton PrismaClient (avoids connection leaks on hot reload).
   - Explicit $connect() on startup to fail fast if DB is unreachable.
   - Explicit $disconnect() on shutdown to release connections.
   - Using global object caching in development for nodemon compatibility.
   - Prisma error format is set to 'pretty' in development for better debugging.
